<objective>
Plan 02 adds the **inline interaction layer** on top of the persistent spine from Plan 01: the Z-Lift Spotlight visual (render layer) and the semantic HTML control dock (UI layer). When a Cell is selected (URL -> selectedCellId), the canvas lifts that Cell's hex in place (scale + glow, neighbors dim, spotlight blooms, focus cone wires it to Core) WITHOUT rebuilding the scene, and a keyboard-navigable HTML dock beside the canvas surfaces the armed session controls (Start/Finish/Cancel) and Core allocation sliders — the same commands the canvas hex tap dispatches. It also adds the Cell-edit Radix Dialog (D-06 configuration exception).

This plan implements D-04 (responsive split — universal Z-Lift, dock position adapts, AND chrome collapses to essentials below md: via a Radix Menu Cell-switcher), D-06 (Cell edit = Dialog; Cell inspect = inline in dock), D-07 (fixed framing + Z-Lift only — object-scale, NO camera/container movement), and D-08 (HTML dock beside canvas — semantic controls, Z-Lift is visual-only). It threads the selectedCellId projection through the adapter into updateFlowgridScene so URL-only selection changes drive the visual.

Purpose: Make the inline module interaction feel like "the module is the interface" while keeping the canvas a visual supplement paired with first-class semantic controls.
Output: Z-Lift scene operation + pure target helper, HTML ZLiftDock component, EditCellDialog, adapter selection threading.
</objective>

**As a** Flowgrid player, **I want to** tap a Cell and have it lift into focus with its controls right beside it on a semantic dock, **so that** I can start a session or tune Core allocation inline without anything covering the living grid.

This plan delivers the inline-interaction slice of the phase story (Plan 01 delivered the persistence spine).
