# Phase 6.1: Canvas-Always-Visible Layout Pivot - Research

**Researched:** 2026-06-27
**Domain:** React Router v7 layout/persistence patterns, PixiJS v8 build-once lifecycle + ticker pause, render↔UI screen-space handoff, responsive IA
**Confidence:** HIGH

## Summary

Phase 6.1 is an **information-architecture pivot**, not a feature phase and not a renderer rewrite. It restructures the React Router v7 route tree so `FlowgridCanvas` (the PixiJS hex lattice + particles) mounts **once in a persistent layout slot** and stays mounted/visible across `/`, `/cells/:id`, and `/core`, with `/settings` and `/forge` rendered as full-screen **overlays above the still-mounted canvas** rather than route swaps. This dissolves the structural blocker that made every particle-emitting event (active-session Current trails, finish Bloom/Activation/Core ripple, Core convert/store ripple, rejuvenation token-grant flash) fire while the canvas was unmounted off-screen — the root cause 06-05 Task 3 could not close.

The locked decisions (D-01…D-09) are already settled in CONTEXT.md. This research resolves the nine **the agent's-discretion** items into concrete, code-grounded recommendations: which React Router v7 pattern to use, exactly how to keep the build-once invariant intact across navigation, how to pause the Pixi ticker during takeovers without destroying the scene, how to add the Z-Lift scene operation within the render-layer boundary, how the render↔UI screen-space handoff works without leaking Pixi refs, how to wire the URL → selection → Z-Lift + dock, the responsive breakpoint strategy, the a11y/WebGL-fail continuity contract, and what the re-opened 06-05 Task 3 acceptance gate must verify.

**Primary recommendation:** Adopt a **pathless React Router v7 layout route** (`element: <AppLayout/>` with no `path`, wrapping all five routes as `children`) that mounts `FlowgridCanvas` + persistent chrome exactly once and renders the route-specific dock/overlay via `<Outlet/>`. Drive the Pixi ticker pause from an explicit `takeoverActive` store flag set from route `handle` metadata (NOT `document.visibilityState` alone — that does not flip for in-DOM overlays). Add the Z-Lift as a new in-place operation on `updateFlowgridScene` keyed off a `selectedCellId` store projection derived from the URL. Keep `main.tsx` **StrictMode-free** (verified — line 32 does not wrap `<App/>`); adding StrictMode would break the build-once invariant in dev via the setup→cleanup→setup double-cycle.

## Architectural Responsibility Map

| Capability | Primary Tier | Secondary Tier | Rationale |
|------------|-------------|----------------|-----------|
| Route tree / persistent canvas slot | Frontend Server (Router) | — | React Router v7 owns the URL↔view map; the layout route is the only tier that can keep a component mounted across URL changes. |
| Pixi Application lifecycle (build-once) | Browser / Client (FlowgridCanvas) | — | `FlowgridCanvas` owns the mount effect; must live in the persistent layout so it never unmounts on navigation. |
| Scene graph + Z-Lift visual | Render (`src/render/flowgrid`) | — | PixiJS scene mutation is the renderer's job; respects the ESLint-enforced no-React/no-Zustand/no-Dexie boundary (`eslint.config.js:131-176`). |
| HTML dock controls (Start/Finish/Cancel, sliders, inspect) | Browser / Client (UI) | — | Per PROJECT.md accessibility rule + D-08: semantic HTML controls are first-class peers; canvas is `role="img"`. |
| Selection projection (selectedCellId) | App store (`src/app/store`) | — | View-state only (mirrors the URL); single source the canvas adapter + HTML dock both read. Never durable. |
| Takeover overlay rendering | Browser / Client (UI) | Router | Takeovers render above the canvas (high z-index); route `handle` declares takeover-ness. |
| Particle emission gate (takeover/reduceMotion) | Render adapter | App store | `connectFlowgridAdapter`'s emit callback already gates on reduceMotion; add `takeoverActive` gate. |

### Locked Decisions

- **D-01:** URL reflects view; canvas persists in a layout slot. Core interactions update the browser URL (tapping a Cell navigates to `/cells/:cellId`, opening Core goes to `/core`). The `FlowgridCanvas` is lifted OUT of the route-swap into a persistent layout slot that route changes do NOT unmount (React Router v7 layout route with `<Outlet />`, or canvas mounted at the App shell above `RouterProvider` — planner picks the idiomatic pattern). The existing route table shape survives (`/`, `/cells/:id`, `/core`, `/settings`, `/forge`); only the canvas mount point moves. URLs stay deep-linkable, back button works naturally, refresh lands in the same view. **Critical constraint:** Phase 6 D-05 build-once — the Pixi Application + scene must be created ONCE and never destroyed/rebuilt across route changes (Pitfall 3 — rebuild-on-dispatch kills particle systems; the same applies to rebuild-on-navigate).
- **D-02:** Takeovers are overlays above the still-mounted canvas. `/settings`, `/forge`, and any future `/history` render as full-screen overlay layers (high z-index) ABOVE the canvas, which stays mounted (hidden, not unmounted). Returning from a takeover is instant — no Pixi re-init, no scene rebuild, no lost ambient Current trail from an active session. D-05 build-once is preserved across takeovers. **Mitigation:** the Pixi ticker must pause while a takeover is active (the canvas is not visible), via `document.visibilityState` and/or an explicit `takeoverActive` flag threaded into the adapter — the planner picks; the goal is "no wasted CPU/battery rendering behind a full-screen overlay, no particle emission into a hidden canvas."
- **D-03:** Home chrome persists across all core views. The `FlowgridHome` chrome — `ReturnCues` stat-chip rail, New Cell button + Radix Dialog, `ResumeSessionPrompt` / `RejuvenationResumePrompt`, and the D-06 always-visible semantic Cell list (`<nav aria-label="Cells">`) — is part of the persistent layout slot, visible at `/`, `/cells/:id`, AND `/core`. The inline Cell/Core view renders in a dedicated zone (the Z-Lift side dock on desktop, bottom strip on mobile — D-04) WITHOUT displacing the chrome. This keeps orientation consistent and the Cell list usable as a quick-switcher across views. The chrome does NOT persist during takeovers (those are full-screen overlays).
- **D-04:** Responsive split — universal Z-Lift, dock position adapts. A single layout adapts across viewports; there is no separate mobile interaction paradigm. **Desktop:** full inline Z-Lift Spotlight (sketch 001) with the control dock in the negative space beside the lifted module. **Mobile (phone):** canvas takes the top portion of the viewport, the Z-Lift control dock reflows to a compact bottom strip (always visible alongside the canvas — NOT a sheet that covers it), and chrome collapses to essentials (Cell-switcher + back control). No sheet, modal, or overlay ever covers the canvas during core play on any device — "effort visible as it happens" holds literally everywhere. Z-Lift (object-scale + dim neighbors + spotlight) happens within the smaller mobile canvas. **Touch:** tap-only parity with desktop click (no gesture vocabulary this phase). The exact breakpoint values and chrome-collapse ordering are agent's discretion (Tailwind responsive classes — likely `sm:`/`md:`/`lg:` thresholds).
- **D-05:** History = layout policy only; defer the dedicated route. Phase 6.1 establishes the POLICY that History is permitted as a full-screen takeover (alongside Settings and Forge build-choice), but does NOT build a new `/history` route. Existing history surfaces continue unchanged: per-Cell completed sessions in the inline Cell view, forge history in `/forge` (`ForgeSummary` + `ForgeChoiceList`), CSV/JSON export in `/settings`. A future phase can add a dedicated `/history` aggregator if users need it; PROD-02 (advanced analytics, trends, long-term insights) stays v2+.
- **D-06:** Cell edit = Radix Dialog; Cell inspect = inline in dock. Editing a Cell (name, color, icon, daily target) opens a Radix Dialog mirroring the existing Cell-create Dialog pattern (Phase 3) — a brief, dismissable modal overlay. This is acceptable because Cell edit is NOT signal-producing (it's configuration), so the "never covered" rule (which protects signal-producing core play) does not apply. Cell inspect details (XP, level, Momentum, Charge, daily milestone progress, Activation state, installed modules, recent sessions — the current `CellBoard` contents) render INLINE in the dock zone, always visible when a Cell is selected. Archive is a button in the inline inspect zone (dispatches `archive_cell`). Module inspection (MOD-02 tiles — level badge, `phase1Behavior`, per-level effect) lives in the Cell inspect inline zone; the Z-Lift dock shows a COMPACT control cluster per sketch 001 (exact density is agent's discretion).
- **D-07:** Fixed framing + Z-Lift only — no pan/zoom camera. v1 ships NO viewport pan or zoom. The grid always fits the viewport (today's fixed-framing behavior — Phase 3 D-04 / Phase 6 D-04). Selecting a Cell relies on Z-Lift (object-scale + dim neighbors + spotlight bloom + focus cone to Core) to focus attention — no camera movement. **No `pixi-viewport` dependency; no custom camera module** (resolves the STACK.md renderer-phase verification flag for this phase — the dependency is not evaluated because it is not needed). If Cell counts grow beyond what fits the viewport in a future phase, pan/zoom can be added then.
- **D-08:** HTML dock beside canvas — semantic controls, Z-Lift is visual-only. The Z-Lift control dock is a React/HTML component rendered OUTSIDE the canvas, positioned beside it (absolute/flex overlay matched to the lifted module's screen position). The canvas Z-Lift animation (lift, glow, dim neighbors, spotlight, focus cone) is PURELY VISUAL FLOURISH; the actual controls — Start/Finish/Cancel session buttons, Core allocation sliders, inspect stats, archive, edit trigger — are semantic HTML (`<button>`, `<input>`, `<label>`, `role`/`aria`). Fully keyboard-navigable (Tab/Enter/Space), screen-reader-friendly, axe-clean. This honors PROJECT.md's accessibility rule literally and extends Phase 6 D-06 (always-visible Cell list) to the inline dock. The canvas keeps `role="img"` (it is a visual supplement; the HTML dock + Cell list are the semantic interface). Tapping a hex on the canvas and the HTML dock button dispatch the SAME command (two paths, one truth).
- **D-09:** Keep the canvas zone on WebGL-fail; show the D-07 message in-place. When WebGL/Canvas init fails, the layout does NOT collapse or reflow — the canvas zone stays in its usual position and size, and the Phase 6 D-07 graceful-degradation note (`role="status" aria-live="polite"` "Visuals unavailable — use the Cell list") renders IN that zone (today's `FlowgridCanvas.tsx` behavior). The HTML dock, Cell list, `ReturnCues`, and inline Core view all stay in their usual spots and remain fully operable. The layout is identical whether WebGL works or not — only the canvas content differs (visuals vs message). No disorienting reflow on failure, no layout shift between machines/reloads.

### Pattern 1: React Router v7 Layout Route (D-01 — RECOMMENDED)

**What:** A **pathless parent route** (`element: <AppLayout/>` with no `path` key) wrapping all five routes as `children`. `AppLayout` mounts `FlowgridCanvas` + persistent chrome exactly once and renders the active child via `<Outlet/>`. Navigating between siblings swaps only the `<Outlet/>` subtree — the layout (and the canvas) stay mounted.

**When to use:** Any time a component must persist across route changes. This is the idiomatic v7 persistent-layout pattern. `[CITED: reactrouter.com/api/components/Outlet]` — Outlet is marked `[MODES: framework, data, declarative]`, confirming it works in the declarative `createBrowserRouter` mode Flowgrid already uses (`routes.tsx:4`). `[CITED: reactrouter.com/start/framework/routing]` confirms: "Child routes are rendered through the `<Outlet/>` in the parent route" and layout routes "create new nesting for their children, but they don't add any segments to the URL."

**Example:**
```tsx
// src/app/routes.tsx — RESTRUCTURED
import { createBrowserRouter } from 'react-router';
import { AppLayout } from '../ui/shell/AppLayout.js';
import { HomeDock } from '../ui/flowgrid-home/FlowgridHome.js';   // was FlowgridHome; now the / child
import { CellDock } from '../ui/cell-board/CellBoard.js';         // was CellBoard; the /cells/:id child
import { CoreDock } from '../ui/core-panel/CorePanel.js';         // was CorePanel; the /core child
import { SettingsTakeover } from '../ui/settings/SettingsPanel.js';
import { ForgeTakeover } from '../ui/forge-panel/ForgePanel.js';

export const router = createBrowserRouter([
  {
    element: <AppLayout />,          // pathless — wraps everything; canvas + chrome persist
    children: [
      { index: true, element: <HomeDock /> },                 // `/`
      { path: 'cells/:cellId', element: <CellDock /> },       // `/cells/:id`
      { path: 'core', element: <CoreDock /> },                // `/core`
      {
        path: 'settings',
        element: <SettingsTakeover />,
        handle: { takeover: true },   // D-02: AppLayout reads this via useMatches()
      },
      {
        path: 'forge',
        element: <ForgeTakeover />,
        handle: { takeover: true },
      },
    ],
  },
]);
```

export function AppLayout() {
  const matches = useMatches();
  // D-02: deepest match's handle.takeover tells us if the canvas is covered

### Pattern 2: Build-Once Across Navigation (D-01 + D-05)

**What:** `FlowgridCanvas`'s mount effect (`FlowgridCanvas.tsx:64-188`) has an **intentionally empty `[]` dependency array** and runs once per component instance. Lifting it into the layout route means the component instance persists across child navigation → the effect runs **once per app session**, and the cleanup path (`FlowgridCanvas.tsx:163-183`) fires **only on full app teardown** (never on route change).

**Verification:** `[VERIFIED: codebase]` — the effect at `FlowgridCanvas.tsx:64` is `useEffect(() => { ... }, [])` with an inline comment (`FlowgridCanvas.tsx:184-191`) documenting the deliberate empty deps. Combined with the Outlet semantics (`[CITED: reactrouter.com/api/components/Outlet]` — the parent stays mounted, only the Outlet child swaps), this guarantees the build-once invariant. The `onCellTap` ref pattern (`FlowgridCanvas.tsx:61-62`) is the template for NOT re-running the effect when callbacks change — extend the same pattern for `selectedCellId` if needed (read latest via a ref or from the store inside the adapter callback).

**StrictMode landmine (CRITICAL):** `[CITED: react.dev/reference/react/StrictMode]` confirms StrictMode "runs one extra setup+cleanup cycle for every Effect" in dev (setup → cleanup → setup). For a build-once Pixi effect this would destroy the Application on first mount then rebuild it — breaking the invariant **in dev only** (production is unaffected). **`[VERIFIED: codebase]`** `main.tsx:32` does NOT wrap `<App/>` in `<StrictMode>`, so this does not occur today. **Planner rule: do NOT add `<StrictMode>` without first making the mount effect idempotent** (e.g., a module-level guard that survives the double-invoke, or moving Pixi init outside React). Document this as an explicit constraint in the plan.

### Pattern 3: Takeover Ticker-Pause (D-02 — RECOMMENDED)

**What:** Pause the Pixi ticker + gate particle emission when a takeover overlay is active, via an **explicit `takeoverActive` store flag** set from route `handle` metadata (Pattern 1). Add `document.visibilitychange` as a **secondary** battery-saving listener for tab-switch.

**Why `visibilityState` alone is insufficient:** `[VERIFIED: reasoning + CITED: pixijs.com/8.x/guides/components/application/ticker-plugin]` — the Pixi docs list "Managing visibility events" as a ticker-pause use case, but `document.visibilityState` only flips to `"hidden"` when the **whole tab** is backgrounded. An in-DOM overlay (Settings/Forge covering the canvas) does NOT change document visibility — the user is still looking at the tab. So `visibilityState` cannot detect a takeover. The explicit flag is load-bearing.

**Hook points (exact files):**
- `flowgrid-store.ts`: add `takeoverActive: boolean` field (default `false`).
- `AppLayout.tsx`: derive `takeoverActive` from `useMatches()` (Pattern 1) and `flowgridStore.setState({ takeoverActive })`.

These are already imported via `motion.ts:61,66` and are **idempotent** per the existing `updateFlowgridScene` reduceMotion gate (`scene.ts:422-426`) — calling them repeatedly is a no-op. `[CITED: pixijs.com/8.x/guides/components/ticker]` confirms `ticker.stop()` "cancels the animation frame" without destroying the Application/scene.

### Pattern 4: Z-Lift Scene Operation (D-07 — render layer, NEW)

**What:** A new in-place scene operation that visually focuses the selected Cell: lift-scale the selected hex (e.g. 1.3× via `tweenScalar` on `hex.scale`), dim non-selected neighbors (tween `hex.alpha` to ~0.4), bloom a translucent spotlight Graphics behind the selected hex, and draw a focus cone (line/triangle) from the cell to Core. All reverses on deselect.

**When to use:** Whenever `selectedCellId` is non-null. Drives NO camera movement (D-07 fixed framing); the selected hex scales, the container stays centered.

**Render-layer boundary compliance:** `[VERIFIED: codebase]` `scene.ts:13-15` documents "this file may import PixiJS + domain types + sibling render modules only. No React, Dexie, Zustand, or DOM." The Z-Lift operation takes `selectedCellId: string | null` as a **plain parameter** (like `localDate`/`reduceMotion` today) — it never imports the store. The adapter/UI threads it in.

**Implementation sketch:**
```typescript
// src/render/flowgrid/scene.ts — extend updateFlowgridScene signature
export function updateFlowgridScene(
  app: Application,
  refs: SceneRefs,
  nextSnapshot: FlowgridSnapshot,
  localDate: LocalDateString,
  reduceMotion: boolean,
  dt: number,
  selectedCellId: CellId | null,   // NEW — threaded from store via adapter
): void {
  // ... existing centering + cell diff logic ...

  // Z-Lift pass (in-place; never destroy+rebuild — Pitfall 3)
  const LIFT_SCALE = 1.3;
  const DIM_ALPHA = 0.4;
  for (const [cellId, view] of refs.cells) {
    const isLifted = cellId === selectedCellId;
    const targetScale = isLifted ? LIFT_SCALE : 1;
    const targetAlpha = (selectedCellId !== null && !isLifted) ? DIM_ALPHA : 1;
    if (reduceMotion || dt <= 0) {
      view.hex.scale.set(targetScale);
      view.hex.alpha = targetAlpha;
    } else {
      view.hex.scale.set(tweenScalar(view.hex.scale.x, targetScale, dt));
      view.hex.alpha = tweenScalar(view.hex.alpha, targetAlpha, dt);
    }
  }
  // Spotlight + focus cone: lazily-created tagged Graphics on refs (or persistent
  // children toggled visible). Position from refs.cells.get(selectedCellId)?.x/y.
  // Spotlight circle behind the hex; cone from cell pos to core {0,0}.
}
```

- **D-07 fixed framing:** NO container/camera movement. Only object-scale. `refs.container.x/y` centering (`scene.ts:277-287`) is untouched.
- **reduceMotion:** snap instantly (no tween) — consistent with `scene.ts:279` precedent.

### Pattern 5: Render↔UI Screen-Space Handoff (D-08 — no Pixi ref leak)

**What:** The HTML dock needs the lifted module's screen position to absolutely-position beside it (desktop) or know it's lifted (mobile bottom strip). The render layer exposes **plain `{x, y}` screen coords** — never Pixi refs.

**How (mirrors the proven `particle-anchors.ts` pattern):** `[VERIFIED: codebase]` `particle-anchors.ts:38-61` already reads `refs.cells.values()` and emits plain `{x, y}` container-local coords with **type-only** `SceneRefs` imports. Apply the same pattern:
1. `FlowgridCanvas` (UI) reads `refs.cells.get(selectedCellId)` → gets container-local `{x, y}`.
2. Converts to screen coords: `screenX = refs.container.x + cellView.x` (the container is centered at `app.screen.width/2`; `cellView.x` is axialToPixel output). Then map Pixi canvas coords → CSS pixels via the canvas DOM rect (`containerRef.current.getBoundingClientRect()`).
3. Store the result as `selectedCellScreenAnchor: { x: number; y: number } | null` in `flowgridStore` (a UI-computed projection, like `selectedCellId`).
4. The HTML dock reads `selectedCellScreenAnchor` and absolutely-positions.

**v1 simplification (acceptable):** Given D-07 fixed framing (cells at predictable ring positions), a **heuristic side-dock** (right negative space on `lg:`, full-width bottom strip below `md:`) without precise per-cell tracking is acceptable for the first slice. Precise anchor tracking is a polish. Recommend: ship the heuristic first, add anchor projection if the dock reads as misaligned.

### Anti-Patterns to Avoid

- **Mounting `FlowgridCanvas` in a child route (today's bug):** re-introduces the unmount-on-navigate blocker that broke 06-05. Canvas MUST live in the layout route or above it.
- **Adding `<StrictMode>` without idempotency:** `[CITED: react.dev/reference/react/StrictMode]` — breaks the build-once invariant in dev via setup→cleanup→setup. `main.tsx:32` is StrictMode-free today; keep it that way until the mount effect is made idempotent.

- **Destroying the scene on takeover (D-05 violation):** `destroyFlowgridScene` (`scene.ts:433`) is ONLY for unmount. Takeover pause = `stopMotion` + emit gate, never destroy.
- **Leaking Pixi refs across the render↔UI boundary:** the dock must receive plain `{x, y}` coords, not `CellView`/`Graphics` objects. Mirror `particle-anchors.ts`'s type-only-import discipline.
- **Routing selection through `dispatch`:** selection is view state. The single durable mutation path (`dispatch.ts`) is for economy commands only. Use `flowgridStore.setState` for the projection.
- **Camera movement for Z-Lift (D-07 violation):** Z-Lift is object-scale + neighbor-dim only. Never move `refs.container` for focus.

## Common Pitfalls

### Pitfall 1: Layout route remounting the canvas on param change
**What goes wrong:** The canvas's mount effect re-runs when navigating `/cells/A` → `/cells/B` (param-only change), tearing down and rebuilding the Pixi Application — killing the build-once invariant.
**Why it happens:** A naive mental model conflates "route changed" with "component unmounted."
**How to avoid:** `[VERIFIED: codebase + CITED: reactrouter.com]` — with a pathless **layout** route, the `AppLayout` (and thus `FlowgridCanvas`) is the **parent** of the param-changing route. Param changes swap only the `<Outlet/>` child (`CellDock`), never the parent. The layout instance persists across all child navigation including `/cells/A` ↔ `/cells/B`. The empty-deps effect (`FlowgridCanvas.tsx:64,188`) therefore runs once. **Verification step:** add an E2E assertion that `window.__flowgridInspect` returns the SAME non-zero cell count (and the `<canvas>` element's identity is unchanged) before and after navigating between two cells.
**Warning signs:** console shows "FlowgridCanvas: Pixi Application.init failed" or the `__flowgridInspect` probe is briefly `undefined` during navigation.

### Pitfall 2: StrictMode double-mount breaking build-once (dev-only)
**What goes wrong:** In dev, the canvas mounts, unmounts (destroying the Pixi app), then mounts again — flicker + wasted init + potential WebGL context leaks.
**Why it happens:** `[CITED: react.dev/reference/react/StrictMode]` — StrictMode runs setup→cleanup→setup for every effect to surface missing-cleanup bugs.
**How to avoid:** `main.tsx:32` is confirmed StrictMode-free (`[VERIFIED: codebase]`). **Do not add `<StrictMode>` in this phase.** If a future phase wants it, the mount effect must first be made idempotent (module-level init guard, or move Pixi init into a module-level singleton outside React). Document this as a hard constraint.
**Warning signs:** two `createFlowgridApplication` calls in the dev console on first load; canvas flicker.

### Pitfall 3: `document.visibilityState` not firing for takeovers
**What goes wrong:** The ticker keeps running (and particles keep emitting into the hidden canvas) during `/settings` or `/forge` because `visibilitychange` never fired.
**Why it happens:** `visibilityState` flips only on tab backgrounding, NOT on in-DOM overlays.
**How to avoid:** Use the explicit `takeoverActive` flag (Pattern 3) as the primary gate. Treat `visibilitychange` as additive-only for tab-switch battery saving.
**Warning signs:** CPU usage stays high while Settings is open; `__flowgridInspect` still reports particles moving behind the overlay.

### Pitfall 4: Z-Lift rebuild triggering Pitfall 3 (particle system reset)
**What goes wrong:** Adding Z-Lift by destroying+rebuilding the selected hex kills the running particle system — the exact failure mode Phase 6 D-05 fixed for dispatch.
**Why it happens:** The reflex to "clear() and redraw" a Graphics when its visual state changes.
**How to avoid:** Z-Lift mutates **in-place** (`view.hex.scale.set(...)`, `view.hex.alpha = ...`) via `tweenScalar`. Never call `view.hex.destroy()` or rebuild during Z-Lift. The spotlight/cone Graphics are created **once** (lazily on first lift, then persisted on `SceneRefs`) and toggled `.visible`, not recreated.
**Warning signs:** active-session Current trail disappears when a Cell is selected.

### Pitfall 5: z-index / stacking-context traps for overlays-above-canvas
**What goes wrong:** The takeover overlay renders BEHIND the canvas (invisible) or the canvas pokes through; or a Radix Dialog (Cell-edit) can't escape the layout's stacking context.
**Why it happens:** Pixi `<canvas>` creates its own stacking context; Tailwind `z-*` + Radix Portal layering interact non-obviously.
**How to avoid:** The takeover routes render via Radix `Dialog.Portal` (or a dedicated overlay `<div>` with `position: fixed; inset: 0; z-index: 50+`). Radix Portal mounts at `document.body`, escaping the layout's stacking context — the Cell-create Dialog already does this (`FlowgridHome.tsx:117` `Dialog.Portal`). For non-Dialog takeovers (Settings/Forge full-screen), use `fixed inset-0 z-50` on the overlay wrapper. The canvas stays at its default z-index (below). Verify with an E2E that the overlay covers the canvas (assert overlay is visible + canvas `__flowgridInspect` still returns counts — canvas mounted but covered).
**Warning signs:** overlay visible but canvas still showing through corners; or overlay invisible entirely.

### Pitfall 6: `useParams` returning `{}` in the layout component
**What goes wrong:** `AppLayout` tries `useParams()` to read `cellId` and gets `{}` — the Z-Lift never engages.
**Why it happens:** `useParams()` returns the params of the route that rendered the calling component. AppLayout is rendered by the pathless layout route (no params).
**How to avoid:** Use `useMatches()` in AppLayout (returns the full chain; find the match with `cellId` in its `params`). `CellDock` (the `/cells/:id` child) can use `useParams()` directly — unchanged from today's `CellBoard.tsx:55`.
**Warning signs:** Z-Lift never triggers; dock shows no Cell selected.

### Pitfall 7: WebGL-fail reflow (D-09 violation)
**What goes wrong:** On WebGL-fail the canvas zone collapses/resizes, shifting the dock + chrome (disorienting layout shift).
**Why it happens:** The `webglFailed` branch returns a different-sized element.
**How to avoid:** `[VERIFIED: codebase]` `FlowgridCanvas.tsx:197-212` already returns the **same className** (`h-[60vh] ... sm:h-[70vh]`) for both the success and fail branches — only the inner content differs. Preserve this exactly when relocating to the layout slot. The dock + chrome read the store (not the canvas DOM), so they're unaffected by canvas state.
**Warning signs:** layout jumps between a WebGL-capable and a headless machine.

### Pitfall 8: Particle anchors stale after resize
**What goes wrong:** After a breakpoint reflow (D-04), particles emit at pre-resize positions.
**Why it happens:** Cached `CellView.x/y` or container centering not refreshed.
**How to avoid:** `[VERIFIED: codebase]` — `buildParticleAnchors` (`particle-anchors.ts:38`) is called **per emit batch** (not cached), reading live `refs.cells`/`refs.routes`. `updateFlowgridScene` re-centers the container on every snapshot change (`scene.ts:277-287`) and `resizeTo: container` (`scene.ts:453`) auto-updates `app.screen`. So anchors are always fresh. **No special resize handler needed** — but add an E2E that resizes the viewport mid-session and asserts particles still emit at sensible positions.
**Warning signs:** particles clustered in a corner after rotating a phone.
