# Phase 6.1: Canvas-Always-Visible Layout Pivot - Context

**Gathered:** 2026-06-27
**Status:** Ready for planning

<domain>
## Phase Boundary

This is the **information-architecture pivot** that Phase 6 Plan 06-05 Task 3 (human smoke) proved is required: the `FlowgridCanvas` (PixiJS hex lattice + particles) must stay mounted and visible during all core signal-producing interactions so effort is perceivable as it happens. Today the canvas mounts only at `/` (`FlowgridHome`); every gameplay event that emits particles (active-session Current trails, finish Bloom/Activation/Core ripple, Core convert/store ripple, rejuvenation token-grant flash) fires on a different route that **unmounts the canvas** — so particles are never visible during their events. This is an IA/navigation problem, not a rendering bug. Plan 06-05's coordinate-space + reduceMotion fixes (commits 174e6eb, d59062d) are correct and stay committed; they simply cannot close the visibility gap until the layout pivots.

**In scope (1 new requirement + re-closure of 2):**
- **UI-08** — The Flowgrid canvas stays mounted and visible during all core signal-producing interactions (active sessions, session finish, Core convert/store, rejuvenation token grants); only Settings, History, and Forge build-choice flows may take over the full screen as explicit returnable exceptions.
- **Re-closure of UI-03** — full-motion visuals must be perceivable during their emitting events (the pivot is what makes 06-05 Task 3 pass).
- **Re-closure of VER-06** — canvas/WebGL smoke checks pass when particles are visible during events.

**It delivers the pivot, not new game capabilities:**
- Restructure the app shell so `FlowgridCanvas` lives in a persistent layout slot that route changes do NOT unmount.
- Move the session/focus view off `/cells/:id` route-swap and onto the canvas, anchored inline to the selected Cell.
- Make Core convert/store happen inline (Core is NOT a takeover).
- Make module interactions inline-on-grid via the **Z-Lift Spotlight** paradigm (sketch 001 winner).
- Establish Settings / Forge build-choice / (policy-only) History as the only full-screen takeover overlays — rendered ABOVE the still-mounted canvas.
- Preserve and strengthen the sacred `open app -> tap Cell -> start session` flow (now literally one tap with no navigation).
- Re-open and pass 06-05 Task 3 human smoke (particles visible during emitting events).

**Out of scope:**
- A dedicated `/history` aggregator route (D-05 — policy only this phase; PROD-02 analytics stay v2+).
- Viewport pan/zoom camera (`pixi-viewport` or custom) — D-07 keeps fixed framing; deferred until Cell counts grow.
- New game capabilities (advanced modules, patch editor, prestige/Memory, sync transport, notifications/widgets, analytics) — v2+, see REQUIREMENTS.md "Deferred After V1."
- Touch gestures beyond tap (long-press inspect, swipe Cell-switch, pinch zoom) — agent's discretion default is tap-only parity with desktop click.
- Sound/audio (no audio infrastructure in v1).
- Rebuilding the Pixi renderer itself — Phase 6 D-01/D-02/D-04/D-05 full-motion work stays; this phase restructures the SHELL around it, not the renderer.
- Cell edit escalation to a takeover surface (D-06 keeps edit as a Dialog).

**Absorbs / re-opens:**
- Re-opens Phase 6 Plan 06-05 Task 3 (the human smoke that FAILED because of this IA disconnect). Task 3 lands as the final acceptance gate of this phase.

## Implementation Decisions

### Shell Architecture & URLs
- **D-01:** **URL reflects view; canvas persists in a layout slot.** Core interactions update the browser URL: tapping a Cell navigates to `/cells/:cellId`, opening Core goes to `/core`. The `FlowgridCanvas` is lifted OUT of the route-swap into a persistent layout slot that route changes do NOT unmount (React Router v7 layout route with `<Outlet />`, or canvas mounted at the App shell above `RouterProvider` — planner picks the idiomatic pattern). The existing route table shape survives (`/`, `/cells/:id`, `/core`, `/settings`, `/forge`); only the canvas mount point moves. URLs stay deep-linkable, back button works naturally, refresh lands in the same view. **Critical constraint:** Phase 6 D-05 build-once — the Pixi Application + scene must be created ONCE and never destroyed/rebuilt across route changes (Pitfall 3 — rebuild-on-dispatch kills particle systems; the same applies to rebuild-on-navigate).
- **D-02:** **Takeovers are overlays above the still-mounted canvas.** `/settings`, `/forge`, and any future `/history` render as full-screen overlay layers (high z-index) ABOVE the canvas, which stays mounted (hidden, not unmounted). Returning from a takeover is instant — no Pixi re-init, no scene rebuild, no lost ambient Current trail from an active session. D-05 build-once is preserved across takeovers. **Mitigation:** the Pixi ticker must pause while a takeover is active (the canvas is not visible), via `document.visibilityState` and/or an explicit `takeoverActive` flag threaded into the adapter — the planner picks; the goal is "no wasted CPU/battery rendering behind a full-screen overlay, no particle emission into a hidden canvas."
- **D-03:** **Home chrome persists across all core views.** The `FlowgridHome` chrome — `ReturnCues` stat-chip rail, New Cell button + Radix Dialog, `ResumeSessionPrompt` / `RejuvenationResumePrompt`, and the D-06 always-visible semantic Cell list (`<nav aria-label="Cells">`) — is part of the persistent layout slot, visible at `/`, `/cells/:id`, AND `/core`. The inline Cell/Core view renders in a dedicated zone (the Z-Lift side dock on desktop, bottom strip on mobile — D-04) WITHOUT displacing the chrome. This keeps orientation consistent and the Cell list usable as a quick-switcher across views. The chrome does NOT persist during takeovers (those are full-screen overlays).

### Mobile Real-Estate Strategy
- **D-04:** **Responsive split — universal Z-Lift, dock position adapts.** A single layout adapts across viewports; there is no separate mobile interaction paradigm. **Desktop:** full inline Z-Lift Spotlight (sketch 001) with the control dock in the negative space beside the lifted module. **Mobile (phone):** canvas takes the top portion of the viewport, the Z-Lift control dock reflows to a compact bottom strip (always visible alongside the canvas — NOT a sheet that covers it), and chrome collapses to essentials (Cell-switcher + back control). No sheet, modal, or overlay ever covers the canvas during core play on any device — "effort visible as it happens" holds literally everywhere. Z-Lift (object-scale + dim neighbors + spotlight) happens within the smaller mobile canvas. **Touch:** tap-only parity with desktop click (no gesture vocabulary this phase). The exact breakpoint values and chrome-collapse ordering are agent's discretion (Tailwind responsive classes — likely `sm:`/`md:`/`lg:` thresholds).

### Inline Depth & Takeover Scope
- **D-05:** **History = layout policy only; defer the dedicated route.** Phase 6.1 establishes the POLICY that History is permitted as a full-screen takeover (alongside Settings and Forge build-choice), but does NOT build a new `/history` route. Existing history surfaces continue unchanged: per-Cell completed sessions in the inline Cell view, forge history in `/forge` (`ForgeSummary` + `ForgeChoiceList`), CSV/JSON export in `/settings`. A future phase can add a dedicated `/history` aggregator if users need it; PROD-02 (advanced analytics, trends, long-term insights) stays v2+. This keeps 6.1 a focused layout pivot and avoids scope creep into a new feature surface.
- **D-06:** **Cell edit = Radix Dialog; Cell inspect = inline in dock.** Editing a Cell (name, color, icon, daily target) opens a Radix Dialog mirroring the existing Cell-create Dialog pattern (Phase 3) — a brief, dismissable modal overlay. This is acceptable because Cell edit is NOT signal-producing (it's configuration), so the "never covered" rule (which protects signal-producing core play) does not apply. Cell inspect details (XP, level, Momentum, Charge, daily milestone progress, Activation state, installed modules, recent sessions — the current `CellBoard` contents) render INLINE in the dock zone, always visible when a Cell is selected. Archive is a button in the inline inspect zone (dispatches `archive_cell`). Module inspection (MOD-02 tiles — level badge, `phase1Behavior`, per-level effect) lives in the Cell inspect inline zone; the Z-Lift dock shows a COMPACT control cluster per sketch 001 (exact density is agent's discretion).
- **D-07:** **Fixed framing + Z-Lift only — no pan/zoom camera.** v1 ships NO viewport pan or zoom. The grid always fits the viewport (today's fixed-framing behavior — Phase 3 D-04 / Phase 6 D-04). Selecting a Cell relies on Z-Lift (object-scale + dim neighbors + spotlight bloom + focus cone to Core) to focus attention — no camera movement. **No `pixi-viewport` dependency; no custom camera module** (resolves the STACK.md renderer-phase verification flag for this phase — the dependency is not evaluated because it is not needed). If Cell counts grow beyond what fits the viewport in a future phase, pan/zoom can be added then. Simplest option; matches sketch 002 Flat Top-Down's "cheapest to build" framing.

### Accessibility & WebGL-Fail Contract
- **D-08:** **HTML dock beside canvas — semantic controls, Z-Lift is visual-only.** The Z-Lift control dock is a React/HTML component rendered OUTSIDE the canvas, positioned beside it (absolute/flex overlay matched to the lifted module's screen position). The canvas Z-Lift animation (lift, glow, dim neighbors, spotlight, focus cone) is PURELY VISUAL FLOURISH; the actual controls — Start/Finish/Cancel session buttons, Core allocation sliders, inspect stats, archive, edit trigger — are semantic HTML (`<button>`, `<input>`, `<label>`, `role`/`aria`). Fully keyboard-navigable (Tab/Enter/Space), screen-reader-friendly, axe-clean. **This honors PROJECT.md's accessibility rule ("canvas visuals must be paired with normal UI controls, semantic labels, and accessible panels") literally** and extends Phase 6 D-06 (always-visible Cell list) to the inline dock. The canvas keeps `role="img"` (it is a visual supplement; the HTML dock + Cell list are the semantic interface). Tapping a hex on the canvas and the HTML dock button dispatch the SAME command (two paths, one truth).
- **D-09:** **Keep the canvas zone on WebGL-fail; show the D-07 message in-place.** When WebGL/Canvas init fails, the layout does NOT collapse or reflow — the canvas zone stays in its usual position and size, and the Phase 6 D-07 graceful-degradation note (`role="status" aria-live="polite"` "Visuals unavailable — use the Cell list") renders IN that zone (today's `FlowgridCanvas.tsx` behavior). The HTML dock, Cell list, `ReturnCues`, and inline Core view all stay in their usual spots and remain fully operable. **The layout is identical whether WebGL works or not** — only the canvas content differs (visuals vs message). No disorienting reflow on failure, no layout shift between machines/reloads.

### the agent's Discretion

The following are mechanical details the user delegated. The agent picks standard, well-tested approaches consistent with prior-phase patterns and documents them in the plan:

- **React Router v7 layout-route structure** — the exact pattern that keeps `FlowgridCanvas` mounted across route changes. Options: (a) a parent layout route with `<Outlet />` that mounts the canvas once and renders child routes in a sibling slot; (b) canvas mounted at `App.tsx` above `RouterProvider` with routes rendering in a sibling `<div>`. Both satisfy D-01; the planner picks the idiomatic React Router v7 pattern. Must respect the layer boundary (render imports no React/Dexie/Zustand; UI owns wiring) and the `FlowgridStoreView` structural contract.
- **Ticker-pause mechanism during takeovers** — how the Pixi ticker is paused while a takeover overlay hides the canvas (D-02). Options: `document.visibilityState` (covers tab-switch too), an explicit `takeoverActive` store flag threaded into the adapter, or both. Must stop particle emission into the hidden canvas (no wasted work) without destroying the scene (D-05).
- **Z-Lift dock positioning** — how the HTML dock tracks the lifted module's screen position so it docks in the right negative space. The render layer exposes the lifted module's screen-space anchor; the HTML dock absolutely positions beside it. Desktop = side dock; mobile = bottom strip (D-04). The render→UI handoff must respect the layer boundary (no Pixi refs leak; screen coords only).
- **Z-Lift selection state ↔ route param wiring** — how selecting a Cell (`/cells/:id`) drives the canvas Z-Lift on that Cell's modules and the HTML dock contents. The store already tracks `activeSession` / `activeRejuvenation`; the planner adds the "selected Cell" / "selected module" projection that both the canvas (visual Z-Lift) and the HTML dock (controls) read.
- **Mobile breakpoint values + chrome collapse ordering** — exact Tailwind breakpoints for the desktop-side-dock → mobile-bottom-strip transition (D-04) and which chrome elements hide first on small screens (Cell list may collapse behind a Cell-switcher button, etc.).
- **Z-Lift dock density / MOD-02 detail** — how much module info (level badge, behavior text, per-level effect from `MODULE_LEVEL_BONUS`) shows in the Z-Lift dock's compact cluster vs the full Cell inspect inline zone. Sketch 001 said "compact control cluster" — the planner picks the split.
- **Canvas `role` attribute** — stays `role="img"` (the canvas is a visual supplement given D-08's HTML dock). Agent confirms; not a user-facing decision.
- **Particle anchor updates on layout resize** — if the canvas layout slot resizes across breakpoints (D-04), `buildParticleAnchors` (Phase 6 06-05 Task 1) must recompute on resize. Agent handles via the existing adapter resize path.
- **Forge-view visual-event miss** — the seed's decision 4 ("accept the miss" for mid-takeover events) means Forge `/forge` flashes may be missed since Forge is a takeover. No replay queue. Agent confirms the emit-gate already handles this (events are transient per Phase 3 D-02; missing them is byte-safe per UI-04).

### Established Patterns
- **Strict layer boundaries** (ESLint-enforced): render imports no React/Dexie/Zustand; UI imports no pixi.js; simulation imports no UI/Pixi/React/Dexie/browser. The Z-Lift visual (render layer), the HTML dock (UI layer), and the selection projection (store layer) must respect these.
- **Build-once, tween in place** (Phase 6 D-05): the Pixi Application + scene are created ONCE and never destroyed across route changes or dispatches. D-01/D-02 extend this to "never destroyed across navigation or takeovers." This is the spine of the pivot.
- **Inline-not-modal** (Phase 3/4/5/6): modals obstruct the Generator. D-06's Cell-edit Dialog is the ONE exception (edit is configuration, not signal-producing). All signal-producing interactions (session, Core, rejuvenation) are inline.
- **Diff for truth, tick for UI** (Phase 3 D-06 / Phase 4 D-04): the canvas tick is cosmetic; durable truth flows through dispatch. The pivot changes only the UI shell, not this contract.
- **Visual events are transient** (Phase 3 D-02 / Phase 6 UI-04): the renderer may drop/reduce/replay/skip them freely; durable state is never affected. Seed decision 4 (accept-the-miss for mid-takeover events) relies on this.
- **Semantic controls as first-class peers** (Phase 6 D-06 + PROJECT.md accessibility rule): the HTML dock + Cell list are the semantic interface; the canvas is a visual supplement. D-08 extends this from the Cell list to the inline dock.
- **Plain string IDs; typed results no throwing; deterministic replay; hybrid records+operation-log; idempotent upsert; integer economy units.** The pivot touches none of these — it's a pure shell restructure.

- **Canvas ↔ persistent layout slot:** `FlowgridCanvas` moves from `FlowgridHome` to the layout route / App shell. Its mount effect runs once per app session.
- **Routes ↔ inline views:** `/`, `/cells/:id`, `/core` all render into the same persistent layout (canvas + chrome), differing only in the dock zone contents. `/settings`, `/forge` render as overlays above.
- **HTML dock ↔ canvas Z-Lift:** the store's selection projection drives both — the canvas lifts the selected module (render layer), the HTML dock renders the controls (UI layer). The render→UI handoff is screen coords only (no Pixi refs).
- **Adapter ↔ takeover-ticker-pause:** `connectFlowgridAdapter`'s emit gate reads a `takeoverActive` signal (or visibility state) to pause particle emission into the hidden canvas.
- **Mobile breakpoint ↔ dock reflow:** the HTML dock switches from side-dock to bottom-strip at the chosen Tailwind breakpoint; the canvas Z-Lift anchors recompute on resize.
- **Dispatch ↔ selection state:** tapping a canvas hex OR a Cell-list link OR a Cell-switcher all set the same "selected Cell" projection → URL updates to `/cells/:id` → canvas Z-Lifts + HTML dock renders. One selection path, three entry points.

## Specific Ideas

- **"Canvas as spine, never covered"** (from the explore seed) is the governing metaphor — every decision extends it. The user's intent is that Flowgrid's identity ("effort becomes visible, useful signal") is betrayed by the current IA where particles fire off-screen; this pivot restores the promise.
- **Z-Lift Spotlight (sketch 001 winner B)** was chosen over Radial Bloom and Command Wheel because it reads as "the module is the interface" most directly while leaving the lattice visible — the lifted module + dimmed neighbors + side dock in negative space keeps the grid as context. The focus cone to the Core visually wires the module into the system.
- **Flat Top-Down (sketch 002 winner A)** was chosen over Tilted Iso Planes and Core Well because it's most readable for "tap the right Cell," cheapest to build, closest to the current renderer, and safest under WebGL-fail. Depth/awe is deferred to v1.1+ if the flat board undersells the promise — but for v1 the flat board ships.
- **HTML dock over Pixi dock (D-08)** was chosen because PROJECT.md's accessibility rule is non-negotiable and UI-02/VER-05 require semantic controls — a Pixi-rendered dock would violate both. The visual cohesion tradeoff (dock isn't "inside" the canvas animation) is accepted; the canvas Z-Lift provides enough visual cohesion.
- **Responsive split over mobile sheet (D-04)** was chosen because a sheet covering the canvas on mobile would betray "effort visible as it happens" on the smallest devices — the very place users most need the reward. The bottom-strip dock is a pragmatic compromise that keeps both visible.
- **Policy-only History (D-05)** was chosen to keep 6.1 a focused layout pivot; a dedicated `/history` aggregator would be scope creep into PROD-02 (v2+ analytics) territory.
- **No pan/zoom (D-07)** was chosen because v1 Cell counts are bounded, Z-Lift already focuses attention, and `pixi-viewport` evaluation is a build-cost risk not worth taking for a scenario v1 may not hit.
