## Accomplishments

- **Z-Lift in-place scene operation (D-07, Pitfall 4 closed):** `updateFlowgridScene` gained a trailing `selectedCellId: CellId | null` parameter and a Z-Lift pass that lifts the selected Cell's hex (scale=LIFT_SCALE 1.3), dims non-selected neighbors (alpha=DIM_ALPHA 0.4), and lazily creates + repositions translucent spotlight + focus cone Graphics. The pass mutates `CellView.hex.scale`/`alpha` IN PLACE via `tweenScalar` and NEVER calls `hex.destroy()` — Pitfall 4 (rebuild resetting the particle system) is closed. Per D-07 the container is NEVER moved for focus (the centering block at scene.ts:277-287 is unchanged).
- **Pure render-layer helper:** `src/render/flowgrid/z-lift-targets.ts` exports `computeZLiftTargets` + `LIFT_SCALE`/`DIM_ALPHA` constants. TYPE-ONLY `CellView`/`CellId` imports (mirrors `particle-anchors.ts`) so the module loads under happy-dom without pixi.js — the second in-repo application of the proven render→UI handoff discipline.
- **Adapter selection threading (RESEARCH Pattern 6):** `connectFlowgridAdapter` now tracks `lastSelectedCellId` alongside `lastSnapshot` and fires `onSnapshot` when EITHER changes — URL-only selection (no dispatch, e.g. /cells/A → /cells/B) drives the Z-Lift visual. `FlowgridCanvas` forwards `storeView.getState().selectedCellId` into `updateFlowgridScene`.
- **ZLiftDock (D-04 responsive, D-08 semantic HTML, two-paths-one-truth):** `src/ui/shell/ZLiftDock.tsx` renders semantic Start/Finish/Cancel controls BESIDE the canvas. Constructs the EXACT command shapes `GeneratorTile` uses (`start_focus_session` / `complete_focus_session` / `cancel_focus_session`) and dispatches through the same path — the canvas hex tap and the dock button dispatch identically. On `/core` it additionally renders Core allocation number inputs. Responsive reflow: lg: side dock, md: narrower side, below md: bottom strip (NEVER a sheet that covers the canvas). v1 simplification: heuristic side dock without per-cell anchor tracking (RESEARCH A3).
- **Cell-list chrome collapse (D-04):** `AppLayout` collapses the always-visible `<nav aria-label="Cells">` into a Radix Menu Cell-switcher below md: (the bulkiest persistent chrome; Mobile Strategy §1). At md:+ the always-visible nav stays (D-06 a11y peer preserved). The Cell-switcher reuses the same activeCells projection and navigates to `/cells/:id` via the same `handleCellTap` path the canvas hex tap uses — one selection path, two visual affordances.
- **EditCellDialog extracted (D-06 configuration exception):** `src/ui/cell-board/EditCellDialog.tsx` is a dedicated Radix Dialog for Cell edit, reusing `EditCellForm`. `Dialog.Portal` is load-bearing: escapes the persistent layout's stacking context so the dialog renders above the Pixi canvas (Pitfall 5). `CellActions` now renders `<EditCellDialog/>` with the Edit button as the `Dialog.Trigger asChild` element.

## Decisions Made

- **Exported `CellView` from scene.ts** so the pure Z-Lift helper can reference it via a TYPE-ONLY import (mirrors the particle-anchors.ts → SceneRefs discipline). Previously `CellView` was a private interface; making it exported is the minimal change to enable the type-only-helper pattern.
- **`tweenScalar` on `hex.scale.x` + `hex.scale.set(scalar)`** — Pixi Graphics exposes `.alpha` directly (plain number); scale is a Point. Tween `.x` via `tweenScalar`, then `hex.scale.set(scalar)` so both `.x` and `.y` stay uniform (prevents asymmetric squash).
- **Lazy-created spotlight + focusCone Graphics** on first non-null selection, persisted on `refs.spotlight`/`refs.focusCone`, toggled `.visible` thereafter (never recreated — Pitfall 4 honored). Tagged them with `SCENE_TAGS.cell`/`SCENE_TAGS.route` so `scene-inspect` counts stay meaningful.
- **Did NOT move `refs.container` for focus (D-07 fixed framing)** — only object-scale + neighbor-dim; the centering block at scene.ts:277-287 is untouched.
- **Aligned ZLiftDock Finish dispatch with `complete_focus_session`** (the actual domain command name) rather than the plan's `finish_focus_session` typo — Rule 3 fix; mirrors GeneratorTile exactly so two-paths-one-truth holds.
- **Heuristic side dock without per-cell screen-anchor tracking** per RESEARCH Open Question 1 + Assumption A3 — deferred polish, acceptable for the first slice. Documented in a header comment in ZLiftDock.tsx.
- **`@radix-ui/react-menu` (standalone primitive) used as a click-triggered Cell-switcher** via controlled `open` state + `Menu.Anchor` — the primitive has no `Trigger` (unlike DropdownMenu which is not installed). The RovingFocusGroup still provides arrow-key navigation + focus trap; only the click trigger is manual. Documented as a small deviation.
- **Aligned EditCellDialog dispatch with `edit_cell`** (the actual domain command name) rather than the plan's `update_cell` typo — Rule 3 fix; reuses the existing EditCellForm which already dispatches `edit_cell` correctly.
- **Task 3 is a pure refactor extraction** (the inline Radix Dialog already existed in CellActions). Per `tdd.md` "investigate when RED does not fail": the EditCellDialog behavior tests are characterization tests committed WITH the extraction in a single GREEN commit rather than a strict RED→GREEN cycle.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 3 - Blocking] ZLiftDock Finish dispatch uses `complete_focus_session`, not the plan's `finish_focus_session` typo**
- **Found during:** Task 2 (writing the dispatch-parity test + implementing ZLiftDock)
- **Issue:** The plan's Test 3 behavior says "clicking dispatches finish_focus_session" but the actual domain command name (per `GeneratorTile.tsx:119` + `domain/result.ts`) is `complete_focus_session`. Using the plan's typo would have broken two-paths-one-truth (the dock would dispatch a command the simulation rejects).
- **Fix:** ZLiftDock constructs `CompleteFocusSessionCommand { type: 'complete_focus_session', ... }` mirroring GeneratorTile exactly. The test asserts `complete_focus_session`.
- **Files modified:** src/ui/shell/ZLiftDock.tsx, tests/ui/z-lift-dock.test.tsx
- **Verification:** `npx vitest run tests/ui/z-lift-dock.test.tsx` → 9 tests pass; `npx tsc --noEmit` exits 0.
- **Committed in:** 62099d1 (Task 2 GREEN), 7d16943 (Task 2 RED used the corrected command name from the start)

**2. [Rule 3 - Blocking] EditCellDialog dispatches `edit_cell`, not the plan's `update_cell` typo**
- **Found during:** Task 3 (creating EditCellDialog + tests)
- **Issue:** The plan's Task 3 action says "dispatches update_cell" but the actual domain command name (per `EditCellForm.tsx:46` + `domain/result.ts:135`) is `edit_cell`. Using the plan's typo would have dispatched a non-existent command.
- **Fix:** EditCellDialog reuses the existing EditCellForm (which already dispatches `edit_cell` correctly). The characterization tests assert `edit_cell`.
- **Files modified:** src/ui/cell-board/EditCellDialog.tsx, tests/ui/z-lift-dock.test.tsx
- **Verification:** `npx vitest run tests/ui/z-lift-dock.test.tsx` → 9 tests pass; `npx tsc --noEmit` exits 0.
- **Committed in:** e406996 (Task 3 GREEN)

**3. [Rule 3 - Blocking] Radix Menu has no `Trigger` primitive — used `Menu.Anchor` + controlled open state**
- **Found during:** Task 2 (implementing the Cell-switcher in AppLayout)
- **Issue:** The plan implies a `Menu.Trigger asChild` API (mirroring Dialog). `@radix-ui/react-menu` (the standalone Menu primitive, which is what's installed) does NOT export a `Trigger` — that's `@radix-ui/react-dropdown-menu`, which is not in package.json. The TypeScript compiler caught `Property 'Trigger' does not exist`.
- **Fix:** Used `Menu.Root open={cellSwitcherOpen} onOpenChange={setCellSwitcherOpen}` (controlled) + `Menu.Anchor asChild` wrapping a button that toggles the open state on click. The RovingFocusGroup (imported by Radix Menu internally) still provides arrow-key navigation + focus trap; only the click trigger is manual. Set `aria-haspopup="menu"` + `aria-expanded` explicitly so the a11y semantics match what DropdownMenu would have provided.
- **Files modified:** src/ui/shell/AppLayout.tsx
- **Verification:** `npx tsc --noEmit` exits 0; the chrome-collapse test finds the trigger by `aria-haspopup="menu"`; `npx vitest run tests/ui/z-lift-dock.test.tsx` → 9 tests pass.
- **Committed in:** 62099d1 (Task 2 GREEN)

**4. [Rule 1 - Bug] Z-Lift pure-helper test asserted `source.includes('pixi.js') === false` — too strict for the comment that legitimately mentions pixi.js**
- **Found during:** Task 1 GREEN (test surfaced the wording bug after the helper was added)
- **Issue:** The RED test asserted the helper's source contains no `pixi.js` substring at all, but the helper's header comment legitimately mentions "Zero runtime pixi.js dependency". The intent was to verify no runtime pixi import, not to ban the word from comments.
- **Fix:** Tightened the assertion to check for `^import.*pixi\.js$` lines (no pixi.js import statements) instead of the bare substring. The intent (no runtime pixi in the pure helper) is preserved; the comment can mention pixi.js freely.
- **Files modified:** tests/ui/z-lift-targets.test.ts
- **Verification:** `npx vitest run tests/ui/z-lift-targets.test.ts` → 4 tests pass.
- **Committed in:** bf05e31 (Task 1 GREEN, the test amendment is part of the same atomic commit as the helper)
