## Shell architecture & URLs

### Q1 — When the user taps a Cell (or opens Core inline), should the browser URL change?

| Option | Description | Selected |
|--------|-------------|----------|
| URL reflects view | Tapping a Cell → `/cells/:id`, Core → `/core`; canvas stays mounted in a persistent layout slot that route changes don't unmount. Takeovers are also routes. Back button works; URLs shareable/deep-linkable; refresh lands in same view. Most "webby"; preserves route table shape. | ✓ |
| Core play on '/' only | Core play lives on '/' as internal view-state; URL stays '/'. Only takeovers (Settings/Forge/History) are real routes. Simplest canvas-persistence, but core views aren't deep-linkable and back exits core play. | |
| Query/hash params on '/' | Core interactions use query params/hash on '/' (e.g. `/?cell=music`) so canvas never unmounts AND view is in URL. Middle ground. | |

**User's choice:** URL reflects view
**Notes:** Preserves the existing route table shape; just lifts the canvas out of the route-swap. Critical constraint: Phase 6 D-05 build-once — Pixi scene created once, never rebuilt across route changes.

### Q2 — When a takeover (Settings/Forge/History) is open, what happens to the canvas?

| Option | Description | Selected |
|--------|-------------|----------|
| Overlay (canvas stays mounted) | Takeovers render as full-screen overlay above the still-mounted canvas. Pixi app/scene/particles never destroyed; returning is instant; D-05 preserved. Ticker pauses while hidden (visibility/flag). | ✓ |
| Replace (unmount canvas) | True route replacement; returning re-mounts + rebuilds. Simpler tree; no ticker-while-hidden. Tradeoff: violates D-05 build-once; every Settings trip pays Pixi re-init; ambient Current trail lost. | |
| Side-drawer (canvas peek) | Canvas stays mounted; takeover is a large side-drawer leaving a sliver visible. Honors "always visible" most literally. Tradeoff: less "full-screen"; sliver may be useless; per-takeover layout complexity. | |

**User's choice:** Overlay (canvas stays mounted)
**Notes:** D-05 build-once preserved. Ticker pause via document.visibilityState and/or takeoverActive flag (agent's discretion).

### Q3 — When in an inline view (/cells/:id, /core), should the home chrome stay visible?

| Option | Description | Selected |
|--------|-------------|----------|
| Chrome persists everywhere | Home chrome (ReturnCues, New Cell, Cell list nav, resume prompts) part of persistent layout across all core views. Inline view renders in dedicated zone without displacing chrome. Consistent orientation; Cell list as quick-switcher. | ✓ |
| Chrome only on '/' | Selecting a Cell hides chrome; inline view gets full zone + back button. Cleaner focus; more room. Tradeoff: less orientation; switching Cells = back to '/' first. | |
| Adaptive chrome | Minimal chrome (Cell-switcher + back) persists in core views; full rail only on '/'. Balance. Tradeoff: two chrome modes (more code). | |

**User's choice:** Chrome persists everywhere
**Notes:** Inline Cell/Core view renders in a dedicated zone (side dock desktop / bottom strip mobile) without displacing chrome.

## Mobile real-estate strategy

### Q1 — How should the always-visible canvas + inline interaction work on mobile phones?

| Option | Description | Selected |
|--------|-------------|----------|
| Responsive split (no sheet) | One layout adapts. Phone: canvas top, compact bottom strip for controls (not a sheet), Z-Lift in smaller canvas, chrome collapses to essentials. No sheet covers canvas. Honors "effort visible" on every device. | ✓ |
| Desktop inline + mobile sheet | Desktop: pure inline Z-Lift. Mobile: bottom sheet over mounted canvas during interaction; retracts after action so burst is visible. Canvas mounted (D-05 preserved) but mostly covered during interaction. | |
| Mobile mirrors desktop | Mobile = full inline Z-Lift with side dock; accept cramped controls. Simplest. Tradeoff: unusable controls on small screens likely. | |

**User's choice:** Responsive split (no sheet)
**Notes:** Universal Z-Lift paradigm; dock position adapts (side desktop / bottom strip mobile). Tap-only parity with desktop (no gesture vocabulary this phase).

### Q2 — Where do Cell edit and Cell inspect details live in the inline world?

| Option | Description | Selected |
|--------|-------------|----------|
| Edit=Dialog, Inspect=inline | Edit = Radix Dialog (mirrors Cell-create pattern). Inspect details (XP, level, Momentum, Charge, milestone, Activation, recent sessions) inline in dock, always visible when Cell selected. Dock focused on reading + frequent actions. | ✓ |

**User's choice:** Edit=Dialog, Inspect=inline

### Q3 — Does v1 need viewport camera movement (pan/zoom), or does fixed framing + Z-Lift suffice?

| Option | Description | Selected |
|--------|-------------|----------|
| Fixed framing + Z-Lift only | Grid always fits viewport (today's behavior). Z-Lift (object-scale + dim + spotlight) focuses attention. No pixi-viewport; no camera module. Works if v1 Cell count bounded. Matches sketch 002 "cheapest." | ✓ |
| Pan-to-Cell (no zoom) | Pan if grid exceeds viewport; selecting off-screen Cell pans it in. Zoom fixed. Needs camera module (evaluate pixi-viewport vs custom). Adds dependency for a scenario v1 may not hit. | |
| Zoom-to-Cell on select | Selecting Cell zooms viewport toward it; Z-Lift in zoomed frame; deselect zooms out. Most game-like; highest cost; mobile/a11y risk. Needs pixi-viewport or custom camera. | |

**User's choice:** Fixed framing + Z-Lift only
**Notes:** No pan/zoom in v1. Resolves STACK.md pixi-viewport verification flag by deferral. Pan/zoom deferred to future phase if Cell count grows.

## A11y & WebGL-fail contract

### Q1 — Is the Z-Lift control dock semantic HTML or Pixi canvas pixels?

| Option | Description | Selected |
|--------|-------------|----------|
| HTML dock (beside canvas) | Dock is React/HTML outside canvas, positioned beside it. Canvas Z-Lift purely visual. Controls are semantic HTML (buttons/forms/labels). Keyboard-navigable, SR-friendly, axe-clean. Honors PROJECT.md accessibility rule. WebGL-fail: dock + Cell list keep working. | ✓ |
| Pixi dock (inside canvas) | Dock rendered inside canvas as Pixi Text/Sprite. Max visual cohesion. Tradeoff: canvas is role=img → NOT keyboard/SR accessible (violates UI-02/VER-05); WebGL failure kills dock. | |
| Hybrid (Pixi + HTML peer) | Pixi visual dock for mouse/touch + separate HTML cluster for keyboard/SR. Max cohesion + a11y. Tradeoff: two control surfaces to maintain; visual duplication. | |

**User's choice:** HTML dock (beside canvas)
**Notes:** Strongest a11y. Canvas stays role="img" (visual supplement); HTML dock + Cell list are the semantic interface. Tapping hex and HTML button dispatch the same command (two paths, one truth). Cascades to resolve most of WebGL-fail too.

### Q2 — When WebGL fails, does the layout keep the canvas zone or collapse it?

| Option | Description | Selected |
|--------|-------------|----------|
| Keep zone, show message | Canvas zone stays in usual position/size; D-07 "Visuals unavailable — use Cell list" note renders IN zone (today's behavior). HTML dock/Cell list/Core view stay in usual spots. Layout identical working/broken — no reflow. | ✓ |
| Collapse zone, expand HTML | Canvas zone collapses on WebGL-fail; HTML surfaces expand. More room when canvas dead. Tradeoff: layout shifts between states; disorienting reflow. | |

**User's choice:** Keep zone, show message
**Notes:** Consistent layout whether WebGL works or not. Today's FlowgridCanvas.tsx D-07 behavior continues. No reflow on failure.

## the agent's Discretion

Areas where the user delegated the mechanical choice to the agent (document in the plan):

- **React Router v7 layout-route structure** — parent layout route with `<Outlet />` vs canvas-at-App-shell-above-RouterProvider. Both satisfy D-01; pick the idiomatic v7 pattern.
- **Ticker-pause mechanism during takeovers** — `document.visibilityState`, explicit `takeoverActive` store flag, or both.
- **Z-Lift dock positioning** — how the HTML dock tracks the lifted module's screen position (render exposes screen-space anchor; HTML absolutely positions beside it; desktop side / mobile bottom strip).
- **Z-Lift selection state ↔ route param wiring** — how `/cells/:id` drives canvas Z-Lift + HTML dock contents (store "selected Cell/module" projection).
- **Mobile breakpoint values + chrome collapse ordering** — Tailwind thresholds; which chrome elements hide first.
- **Z-Lift dock density / MOD-02 detail split** — compact cluster in Z-Lift dock vs full MOD-02 tile in Cell inspect inline zone.
- **Canvas `role` attribute** — stays `role="img"` given D-08 (visual supplement). Confirm.
- **Particle anchor resize recompute** — `buildParticleAnchors` on layout resize across breakpoints.
- **Forge mid-takeover miss** — confirm the emit-gate handles it (events transient per Phase 3 D-02; byte-safe per UI-04).
