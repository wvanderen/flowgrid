## Decisions Made

- **Pathless layout route over canvas-at-App-shell-above-RouterProvider (RESEARCH Pattern 1 ranking):** the layout route is strictly more idiomatic — the router owns the full layout tree, route handle/useMatches stay usable for takeover detection, and deep-link/back/refresh continue to work natively. Canvas-at-shell would lose router awareness of the canvas zone.
- **Explicit `takeoverActive` store flag over `document.visibilityState` alone (RESEARCH Pattern 3 + Pitfall 3):** `visibilityState` does NOT fire for in-DOM overlays (the doc stays "visible" while Settings covers the canvas). The flag is load-bearing; `visibilitychange` would be additive-only for tab-switch battery saving and was deferred this phase.
- **Separate subscribe listener for ticker-pause:** rather than folding the ticker-pause into connectFlowgridAdapter's handler, a second storeView.subscribe listener owns the stopMotion/startTicker calls. This keeps the adapter's emit-drain path focused and makes the takeover gate independently testable. Both calls are idempotent so firing on every store change is safe.
- **Fixed inset-0 z-50 wrappers (not Radix Portal) for takeovers:** SettingsTakeover + ForgeTakeover are not Radix Dialogs, so Radix Portal isn't applicable. The fixed positioning + high z-index achieves the same document-body-level layering per RESEARCH Pitfall 5 — the canvas stays at its default z-index (below), the overlay covers it.
- **`useMatches` over `useParams` in AppLayout (RESEARCH Pitfall 6):** the pathless layout route has no params, so `useParams()` in AppLayout returns `{}`. `useMatches()` returns the full chain with each match's `params` populated for the deepest matching route, so AppLayout can find the `/cells/:id` match's `cellId` param. CellBoard (the /cells/:id child) continues to use `useParams()` directly — unchanged.
- **main.tsx stays StrictMode-free (RESEARCH Pitfall 2):** adding `<StrictMode>` would run the build-once Pixi mount effect through setup→cleanup→setup in dev, destroying the Application on first mount then rebuilding it. The mount effect must first be made idempotent (module-level init guard or moving Pixi init outside React) before StrictMode is safe. Documented as a hard constraint in AppLayout's header comment.
