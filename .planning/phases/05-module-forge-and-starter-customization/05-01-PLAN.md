## Phase Goal

**As a** Flowgrid player **I want to** turn my earned Module Tokens (or Energy) into curated module-level upgrades through a deterministic Forge roll **so that** my focused effort translates into tangible, replayable build choices that persist as real history.

## Objective

Plan 05-01 builds the simulation truth for Phase 5: the atomic `run_forge` command handler, the pure `forgeChoices` reveal selector, the `MODULE_LEVEL_BONUS` content table that makes `ModuleInstance.level` mechanically meaningful, the widened `ForgeHistoryRecord`, the four per-level system reads (Generator / Charge Core / Output / Bloom), the new level-cap invariant, and the unit + property test suites that prove them.

This plan delivers the FULL vertical slice at the simulation layer: after it ships, a developer can drive the entire Forge loop through `runSimulationCommand` in tests and observe atomic payment → reveal-validation → level-up → history-append → event-emission, with deterministic replay and VER-02 invariants holding. Plans 05-02 (persistence migration + Zod) and 05-03 (UI route + chip + tile) build on this truth unchanged.

Purpose: Establish the durable truth and invariants for the Forge loop before any persistence or UI lands — mirroring the Phase 4 wave-1 pattern (`04-01-PLAN.md`) that kept simulation truth independently verifiable.
Output: 6 new files + 12 modified files + 3 new test files; `npm test` green; the `run_forge is not implemented until Phase 5` stub string no longer present.
