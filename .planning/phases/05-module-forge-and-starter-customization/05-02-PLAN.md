## Phase Goal

**As a** Flowgrid player **I want to** reload the app after a Forge roll and see my upgraded modules + full forge history intact **so that** my build choices are durable local-first records, not ephemeral session state.

## Objective

Plan 05-02 ships the persistence layer for the widened ForgeHistoryRecord: the Dexie v3→v4 schema bump (full store set + extracted transform + .upgrade() even on the empty store), the widened Zod `forgeHistorySchema` + satisfies drift guard (Pitfall 6), the export/import envelope updates, and a migration-harness fixture that proves the transform.

The migration is low-risk because the `forgeHistory` store is empty pre-Phase-5 (the Phase 1 `run_forge` stub never wrote rows), but the schema version MUST still bump and the full store set MUST repeat (Dexie requirement, RESEARCH Pitfall 5). This plan is a near-clone of the Phase 4 v2→v3 migration (`04-02-PLAN.md` / `upgradeCoresV2ToV3`).

Purpose: Make the widened record shape durable across reloads and importable across archives before any UI dispatches the new command — mirrors the Phase 4 wave-2 pattern.
Output: 4 modified source files + 1 modified test file; `npm test` green; the new schema version is 4.
