# Continue Here — Phase 06.2 Re-plan

## Why execution stopped

Plan `06.2-01` was built on **factually incorrect git evidence** about commit `29e32a7`
("main view redesign"). The plan assumed `29e32a7` was a single-hunk regression that wrapped
`<Outlet/>` in a hidden div, and that removing the wrapper would restore the known-green
`c26c5ad` state to 14/14 Playwright.

**Empirical runtime testing during execution disproved this.** Execution aborted at Task 1
(Rule 4 architectural checkpoint). User decision (2026-06-29): **keep the redesign — do NOT
revert `29e32a7`.** Route back to `/gsd-plan-phase 06.2` for a corrected plan.

## Corrected git evidence (must inform the new plan)

`git show --stat 29e32a7` → **39 files changed, +1949/-402 lines.** This was a large
intentional product redesign, not a single-hunk regression. Highlights:

- `src/ui/shell/AppLayout.tsx` (116 lines): rewrote `<section>` to `xl:h-screen xl:overflow-hidden`;
  moved canvas into a `relative z-10 ... flex-1` container; absorbed FlowgridHome's chrome into a
  floating surface; **removed `ArchivedCellsFilter`**.
- `src/ui/shell/ZLiftDock.tsx` (673 lines expanded): now the primary interaction surface for
  `/cells/:id`.
- Renderer work: `scene.ts` (+221), new `ambient-current.ts`, `route-anchors.ts`, `particles.ts`.
- New `.impeccable/` design docs, `DESIGN.md`, `PRODUCT.md`.

**Architectural intent of the redesign:** canvas as full-viewport ambient background + floating
ZLiftDock/chrome on top. CellBoard / FlowgridHome / CorePanel are effectively **retired** — the
hidden `<Outlet/>` wrapper was one of multiple retirement mechanisms, not an isolated bug.

## Empirical runtime results (Playwright against production build)

| Variant | Result | Failure mode |
| ------- | ------ | ------------ |
| Bare `<Outlet/>` (plan's literal fix) | **3 passed / 11 failed** | CellBoard mounts but `<canvas width="1280" height="720">` from the `relative z-10 flex-1` container intercepts pointer events on CellBoard's "Return to Flowgrid" link (`release-flow.spec.ts:30`). Error: *"element is visible, enabled and stable … `<canvas>` subtree intercepts pointer events."* |
| `<Outlet/>` in `<div className="relative z-20">` | worse | On `/`, FlowgridHome (via Outlet) renders above the floating chrome, covering the "active Cell" text (`release-flow.spec.ts:21` fails before createCell). |

**Conclusion:** the redesign's canvas-as-full-background + floating-chrome layout is fundamentally
incompatible with visible+clickable child-route content via `<Outlet/>`. No minimal CSS-only fix
reconciles them.

## Recommended re-plan direction (advisory, not binding)

Given the redesign is fixed, the realistic paths are:

- **Option C — Hybrid layout under the redesign:** restructure AppLayout so any child-route content
  the E2E still exercises renders in a visible, non-overlapping region (e.g. a scrollable panel
  beside/below the canvas, or floating panels at z-30 that don't cover home chrome). Preserves the
  redesign; largest execution effort; needs a UX decision about where that content lives.
- **Option B (now legitimate) — Rewrite E2E specs to drive through ZLiftDock:** the original
  plan's prohibition against touching E2E specs was premised on the wrong git evidence; with the
  redesign fixed, the old E2E assertions are testing retired components. Smaller code diff, but
  requires re-deriving what UI-03/UI-08/VER-04/05/06 coverage should look like under the new
  architecture.

The next planner should pick one explicitly and re-derive must_haves from it.
