## Recommended B-01 Fix — Option (a), bare `<Outlet/>` (CONFIRMED)

### Why option (a) is correct (git evidence)

The audit offered three options. **Only option (a) is correct**; the other two are red herrings:

- **Option (a) — remove the wrapper:** ✅ **Correct.** Restores the exact pre-regression state. The git diff of `29e32a7` (`git show 29e32a7 -- src/ui/shell/AppLayout.tsx`) shows the ONLY change to the `<Outlet/>` block was wrapping it:

  ```diff
  -      <Outlet />
  +      <div hidden={!takeoverActive}>
  +        <Outlet />
  +      </div>
  ```

  Before `29e32a7`, `<Outlet/>` was bare (the last child of the section). The E2E suite passed at `c26c5ad` (06-27) with that structure. Removing the wrapper exactly reverses the regression.

- **Option (b) — rewrite E2E tests to drive through ZLiftDock:** ❌ **Wrong.** The audit author overlooked that `release-flow.spec.ts:30` clicks "Return to Flowgrid" — a link **only CellBoard renders** (`src/ui/cell-board/CellBoard.tsx:91`), never ZLiftDock. Rewriting tests to avoid CellBoard would mean deleting the load-bearing assertions. The tests were green at `c26c5ad` precisely because CellBoard was visible; the fix is to restore that visibility.

- **Option (c) — conditionally render Outlet only on takeovers:** ❌ **Wrong.** Makes CellBoard/CorePanel unmount (worse than hidden — breaks React state + the canvas-persistence tests). Also contradicts the E2E tests which require CellBoard visible on `/cells/:id`.

### Strict-mode-violation concern (resolved)

A reasonable worry: if both ZLiftDock AND CellBoard render simultaneously (both have "Start Focus Session" buttons), would `getByRole('button', { name: 'Start Focus Session' })` strict-mode-violate?

**Empirically no** — the E2E suite was green at `c26c5ad` when ZLiftDock (added by `62099cd1` in 06.1-02) AND CellBoard were BOTH visible. The `canvas-smoke.spec.ts` tests already scope to `page.getByLabel('Cell Board for X')` (see comments at `tests/e2e/canvas-smoke.spec.ts:49-59, 257-261`), and the unscoped locators in `release-flow.spec.ts` / `accessibility.spec.ts` evidently resolved cleanly (Playwright may resolve to the first match in DOM order, or the redesign's repositioning of ZLiftDock to an absolute overlay at `xl:` keeps it outside the default tab/heading flow). The load-bearing fact is the green run at `c26c5ad`; option (a) restores that exact state.

### Why vitest passes but Playwright fails (load-bearing context)

Testing Library (`getByRole`) finds hidden elements; Playwright's `toBeVisible()` / `click()` enforce CSS visibility and time out at 30s. This is why the regression slipped past the vitest suite (276/276 green) and was only caught by the Playwright re-run. **The Playwright suite is the only gate that catches B-01-class regressions** — the planner MUST treat `npx playwright test` as the primary acceptance gate, not `npm test`.

## Wiring Plan for W-01 (DATA-07)

### Current state (the gap)

**`src/persistence/database.ts:292-300`** constructs a typed `PersistenceError` and discards it:

```typescript
this.on('blocked', () => {
  const _blockedError: PersistenceError = {
    kind: 'blocked_upgrade',
    message: 'IndexedDB upgrade is blocked, likely by another open tab.',
    recoverable: true,
  };
  // Phase 3 will route _blockedError to the UI error channel.
  void _blockedError;
});
```

The comment "Phase 3 will route" was never fulfilled. `grep blocked_upgrade src/ui/ src/app/` returns **0 matches** (verified — only `src/persistence/` and `tests/persistence/` contain the literal).

### Architecture constraint (load-bearing)

`AGENTS.md` + `tests/persistence/boundaries.test.ts` forbid `src/persistence/` from importing `react`, `zustand`, `../app/*`, etc. **Cannot short-circuit** by importing `flowgridStore` into `database.ts`. The error must cross the boundary via the app layer subscribing to a persistence-tier event.

### Chosen seam: `repository.onBlockedUpgrade(handler)`

The repository already wraps the Dexie instance (`src/persistence/repository.ts:82-90` — holds `private readonly db`, wraps `open()`/`close()`). Adding a subscription method there is the natural seam: persistence owns the Dexie event + error factory; app owns the store mutation.
