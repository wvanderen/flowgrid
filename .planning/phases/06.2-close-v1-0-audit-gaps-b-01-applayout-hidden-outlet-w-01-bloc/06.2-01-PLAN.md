## B-01 Resolution Decision: Option B — Rewrite E2E specs to drive through the redesign's visible chrome

### Choice
**Option B.** Rewrite the three Playwright E2E specs (release-flow, accessibility, canvas-smoke) so every assertion that previously targeted the retired CellBoard / FlowgridHome / CorePanel child routes instead drives through the redesign's always-visible surfaces: the **ZLiftDock** (`CellInspectorDock` on /cells/:id, `CoreInspector` on /core) and **AppLayout's own floating chrome** (the "Flowgrid" logo link, "New Cell" button, Cell-list `<nav aria-label="Cells">`). Leave `src/ui/shell/AppLayout.tsx` and its hidden `<Outlet/>` wrapper **completely untouched** — the wrapper is a deliberate retirement mechanism per the user decision, not an isolated bug.

### Rationale (grounded in the actual current code)
1. **ZLiftDock already has full control parity.** Reading `src/ui/shell/ZLiftDock.tsx` confirms `CellInspectorDock` (lines 136-328) renders the Cell name in an `<h2>`, Start/Finish/Cancel focus buttons, SessionSummary, ModuleChip grid, recent sessions, and CellActions. `CoreInspector` (lines 361-575) renders Convert %/Store % inputs, Apply Allocation, Start/Finish Rejuvenation, Boost, and the Forge link. These are the EXACT controls the E2E needs — the "two-paths-one-truth" principle (D-08) documented in `canvas-smoke.spec.ts:52-55` means ZLiftDock dispatches the same command shapes the retired CellBoard/CorePanel did.
2. **AppLayout's chrome provides every navigation surface.** Reading `src/ui/shell/AppLayout.tsx` confirms the header has a "Flowgrid" `<Link to="/">` (line 146), a "New Cell" button inside a Radix Dialog (line 188), and a Cell-list `<nav aria-label="Cells">` with per-Cell `<Link to={/cells/${id}}>` entries (lines 223-240). All are visible whenever `!takeoverActive`. These replace the retired FlowgridHome/CellBoard navigation links.
3. **Option C would conflict with the redesign's intent.** The user decision states CellBoard/FlowgridHome/CorePanel are "effectively retired." Option C (hybrid layout) partially un-retires them by rendering their content in a visible panel, creating dual-surface redundancy (two sets of Start/Finish buttons) and the strict-mode-violation risk the original research flagged. Option B embraces the retirement.
4. **Substantive coverage is preserved, not deleted.** The rewrite changes WHICH element the locators target, not WHAT the tests verify: VER-04 still drives the full create→focus→outcome→allocation→rejuvenation→forge→reload flow; VER-05 still does keyboard-only + per-route axe scans; VER-06 still probes the canvas scene graph + pixel variance + reduced-motion economy-equivalence; UI-03 still asserts the probe stays non-zero throughout the session lifecycle; UI-08 still asserts canvas persistence across routes + param-change identity + takeover coverage. The canvas probe (`window.__flowgridInspect`) is independent of CellBoard and remains the load-bearing assertion.
5. **Smallest, safest diff.** Zero production code changes for B-01 (AppLayout untouched). Only the three E2E spec files change. No UX decisions, no layout restructuring, no risk of breaking the redesign's visual composition.
</decision>
