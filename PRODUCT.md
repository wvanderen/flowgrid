# Product

## Register

product

## Users

Flowgrid is for people who want their attention, effort, and recovery to become visible structure instead of another productivity ledger. They open it in the middle of ordinary life, often with limited energy, and need the first useful action to be obvious: choose a Cell, start a real session, and return later without punishment.

Users are managing life domains such as Music, Fitness, Writing, Home, Rest, Study, and Social. They need the interface to support focus, return, reflection, and build choices without burying the sacred Generator interaction under configuration or analysis.

## Product Purpose

Flowgrid turns real effort into a playable local-first system. Focus sessions create Current, Charge, Energy, Module Tokens, and build opportunities across a hex lattice of Cells and a central Core.

Success means a user can tap a Cell, do a real thing, and feel that effort become useful signal in a system that is powerful, forgiving, and mechanically legible. Flowgrid should not become a generic task manager, dashboard, social tracker, or passive idle game.

## Brand Personality

Focused systems: calm, precise, quietly magical.

The product should feel like a living instrument for attention: structured enough to trust, responsive enough to feel alive, and restrained enough that the user's real work remains the point. Game feedback should make effort perceptible, but never turn the app into noise.

## Anti-references

- Generic productivity dashboards that foreground charts, task tables, streak pressure, or shame-coded catch-up language.
- Overstimulating game UIs that make the reward layer louder than the real action the user came to do.
- Passive idle-game economies where offline accumulation matters more than effort and recovery.
- Canvas-only interfaces that hide critical actions from keyboard, screen reader, or normal form workflows.
- Marketing-site visual tropes in the app shell: oversized hero sections, decorative card grids, tiny uppercase eyebrows, gradient text, and ornamental motion.

## Design Principles

1. Protect the Generator.
   The path `open app -> tap Cell -> start session` is the primary product promise. Every layout, route, animation, and panel must keep that path easy to find and hard to break.

2. Make effort visible, not performative.
   Current, Charge, Energy, Bloom, Integration, and Forge outcomes should be perceivable as signal movement and state change. Feedback should clarify consequence without exaggerating importance.

3. Keep the system legible.
   Flowgrid can be deep, but each screen should expose the next useful action and explain state through normal controls, labels, and inspections. Simulation owns truth; UI explains and dispatches.

4. Make return powerful and forgiving.
   Missed time should surface stored opportunity, partial progress, and recovery paths. Avoid streak-loss drama, failure framing, or language that makes rest feel like a defect.

5. Let the lattice feel alive without taking over.
   Pixi visuals should carry the emotional and spatial identity of the Flowgrid, while React UI handles controls, forms, history, settings, and accessible inspection with consistent product vocabulary.

## Accessibility & Inclusion

Target WCAG AA for product UI. Canvas visuals must have semantic non-canvas controls for critical actions, accessible selected-state and status surfaces, keyboard paths, and readable contrast.

Reduced motion must be respected. Motion should communicate state changes such as Current movement, Bloom, Core conversion, progress, and completion, with non-animated alternatives that preserve meaning. Return cues and recovery copy should avoid shame, punishment, or moralizing language.
